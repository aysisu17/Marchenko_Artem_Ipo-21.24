
# _Контрольные вопросы лаба №1:_ 

1. Что такое СКВ?
2. Какие проблемы решает Git?
3. Для разработки какой операционной системы используется Git?
4. Что такое репозиторий?
5. В чем разница между GoogleDrive/DropBox/YandexDisk и репозиторием Git?

Ответ: 
1. СКВ (система контроля версий) - это по, которое фиксирует изменения в файлах проекта со временем. Оно позволяет просматривать историю изменений, возвращаться к предыдущим состояниям кода и работать над одним проектом нескольким людям одновременно.
2. - **Потеря данных:** можно восстановить любую предыдущую версию файла.
    
- **Конфликты при совместной работе:** позволяет нескольким разработчикам менять один и тот же код, не затирая правки друг друга.
    
- **Эксперименты:** с помощью «веток» можно пробовать новые идеи, не ломая основной (рабочий) код.
    
- **Прозрачность:** всегда видно, кто, когда и зачем внес конкретное изменение.

3. Git был создан Линусом Торвальдсом специально для разработки ядра **Linux**.
4.  **Репозиторий** — это своего рода хранилище или база данных проекта. В нем содержится не только текущая версия файлов, но и вся история их изменений (коммиты).
5. **GoogleDrive / Dropbox / YandexDisk** — это «облачные флешки». Они просто синхронизируют последнюю версию файла. Если два человека одновременно изменят один файл, сервис создаст «копию, в которой возник конфликт», и вам придется разбираться вручную.
**Репозиторий Git** — это «машина времени». Он хранит не просто файлы, а **историю осмысленных изменений** (коммитов).

# _Контрольные вопросы лаба №2:_

1. Что такое GitBash?
2. Для чего нужна команда ls?
3. Как сменить директорию?
4. Как отобразить текущую директорию?
5. Для чего задаются настройки Git?
6. Как применить одинаковые настройки для всех репозиториев?

Ответ: 
1. **Git Bash** — это приложение для Windows, которое эмулирует среду командной строки Bash. Оно позволяет использовать команды Unix (Linux) и инструменты Git в интерфейсе командной строки.
2. Команда `ls` (list) используется для **просмотра содержимого текущей папки**. Она выводит список всех файлов и поддиректорий.
3. Командой cd (cd путь_папки)
4. Командой pwd (print working directory). Она выводит полный путь к папке, в которой вы сейчас находитесь.
5. Настройки (конфигурация) нужны, чтобы Git понимал, **кто вносит изменения**. Самые важные параметры — `user.name` и `user.email` — они прикрепляются к каждому вашему коммиту, чтобы другие разработчики знали автора правок.
6.  `--global` при вводе команды. Такие настройки сохраняются в вашем профиле пользователя и действуют на все проекты.

# _Контрольные вопросы лаба №3:_

1. Как отобразить текущий статус репозитория?
2. Какие файлы называются Untracked files?
3. Как передать файл под контроль Git?
4. Как отобразить список изменений в репозитории?
5. Как отменить коммит и почему это делать не рекомендуется?
6. Как Git идентифицирует все этапы изменения?

Ответ:
1. Команда `git status`. Она показывает, какие файлы изменены, какие подготовлены к сохранению, а какие Git еще не отслеживает.
2. Это **неотслеживаемые файлы**.
3. Команда `git add название_файла`
4. Команда `git log`
5. Команда `git reset HEAD`. Если вы уже отправили коммит в общий удаленный репозиторий, удаление его из истории вызовет ошибки у других разработчиков. Это нарушает целостность общей истории проекта.
6. Git использует хеш-суммы (SHA-1). Это уникальный буквенно-цифровой код, который создается автоматически для каждого коммита на основе содержимого файлов и метаданных. По этому ID можно найти любое состояние проекта в прошлом.


# _Контрольные вопросы лаба №4:_

1. Как отправить изменения на удаленный репозиторий?
2. Как влить изменения в локальную ветку из удаленного репозитория?
3. Какой командой можно зафиксировать изменения в репозитории?
4. Как используется Gihub в реальной работе?
5. Опишите принцип коллективной работы с Github

Ответ:
1. С помощью команды `git push`
2. С помощью команды `git pull`
3. git commit -m "название коммита"
4. 
- **Хранение кода:** как основной бэкап и центральное хранилище.
    
- **Code Review:** проверка кода коллегами перед тем, как он попадет в основной проект.
    
- **Issue Tracking:** ведение списка задач, багов и планов (задач-тикетов).
    
- **CI/CD:** автоматическое тестирование и деплой кода на сервера.

5.  В основе лежит механизм **Pull Requests (PR)**:
- **Ветвление:** Каждый разработчик создает свою ветку для конкретной задачи (`feature branch`).
    
- **Работа:** Разработчик делает коммиты и отправляет ветку на GitHub (`push`).
    
- **Pull Request:** Создается запрос на слияние своей ветки с основной веткой проекта.
    
- **Обсуждение:** Коллеги смотрят код, оставляют комментарии и, если всё хорошо, подтверждают (Approve).
    
- **Слияние (Merge):** Изменения вливаются в основную ветку, и задача считается выполненной.

# _Контрольные вопросы лаба №7:_
1. Какая ветка создается сразу после создания репозитория?
2. Как задать новую ветку?
3. Что такое head и какая взаимосвязь между git checkout?
4. Как произвести слияние веток?
5. Что значит fast-forward?

Ответ:
1. Ветка master
2. Командой git branch `название новой ветки`
3. HEAD - это указатель на текущий активный коммит или ветку, определяющий состояние вашего рабочего каталога. Команда `git checkout` — это инструмент для управления HEAD.
4. Команда git merge `имя ветки`
5. Fast-forward (FF) - тип слияния веток, при котором Git просто переносит указатель текущей ветки вперед к последнему коммиту объединяемой ветки, не создавая новый коммит слияния.
